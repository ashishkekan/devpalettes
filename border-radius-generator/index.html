What Is a CSS Border Radius Generator?

A CSS border-radius generator is a visual design tool that lets you shape the corners of HTML elements by dragging sliders instead of typing numeric values into a stylesheet. The border-radius property controls how much each corner of an element is rounded—and when you give all four corners different percentage values, the element stops looking like a rectangle and starts looking like an organic blob or abstract shape.

The property accepts up to four values in clockwise order: top-left, top-right, bottom-right, bottom-left. At first glance this seems simple—set all four to 8px and you get a slightly rounded card. But when you start pushing values to 30%, 70%, or mixing them asymmetrically, the element transforms into shapes that would normally require an SVG path or a PNG image. That is the real power of this property, and it is the reason blob backgrounds have become so common in modern web design without any actual image assets.

This generator gives you four independent sliders—one per corner—plus size and color controls. The preview updates as you drag. When the shape looks right, you copy a single border-radius declaration and paste it into your project. Whether you are building a color palette showcase, a SaaS landing page, or a personal portfolio, rounded corners and blob shapes add visual softness that plain rectangles cannot achieve.

Why Use a Border Radius Generator Instead of Writing Values Manually?

Writing border-radius: 8px is trivial. Writing border-radius: 30% 70% 70% 30% / 30% 30% 70% 70% and knowing what it looks like before you see it—that is a different story. The four-value shorthand is not hard to remember, but the relationship between those numbers and the resulting shape is not intuitive. A 60% corner does not look twice as round as a 30% corner. The curves interact with each other in ways that are difficult to predict mentally, especially when the element is not a perfect square.

A generator eliminates that unpredictability. You drag a slider and immediately see whether 55% or 65% looks better for the top-right corner. You can try the Random button to discover shape combinations you would never have thought to type. You can fine-tune a blob until it has exactly the right amount of asymmetry—not too geometric, not too chaotic.

There is also a practical accuracy benefit. When you type four percentage values by hand, it is easy to transpose two numbers or forget a space between them. The generator outputs syntactically clean CSS every time, which matters when you are pasting into a production stylesheet and do not want to debug a missing comma at 2 AM.

How Border Radius Values Work

When you provide four values to border-radius, they apply in a specific clockwise order:

  • First value (Top-Left): Rounds the top-left corner. Higher values create a larger curve that eats into the adjacent edges.
  • Second value (Top-Right): Rounds the top-right corner. When this differs significantly from the first value, the top edge becomes asymmetric.
  • Third value (Bottom-Right): Rounds the bottom-right corner. Pairing this with a contrasting top-left value creates the classic "leaf" or "blob" shape.
  • Fourth value (Bottom-Left): Rounds the bottom-left corner. The interplay between this and the top-right value controls the overall balance of the shape.

Pixel values create fixed-radius corners, while percentage values scale relative to the element's dimensions. This distinction matters enormously: a 12px radius looks proportional on a 300px-wide card but overwhelming on a 40px-wide badge. Percentage values avoid this problem because they scale with the element. At 50% on all four corners of a square, the element becomes a perfect circle. If you need to convert color formats for your shape's background, our converter handles HEX, RGB, and HSL.

There is also an eight-value syntax that most developers rarely use: border-radius: 30% 70% 70% 30% / 30% 30% 70% 70%. The values before the slash control horizontal radii, and the values after control vertical radii. This lets you create independently elliptical curves on each corner, producing even more complex shapes than the four-value syntax allows.

  2. Adjust individual corners: Drag each slider slowly and watch how the shape responds. Pay attention to how changing one corner affects the adjacent edges—the curves share control points, so moving one corner shifts the curve of its neighbors.

Common Use Cases in Modern UI Design

  • Hero section decorations: Large, semi-transparent blobs positioned behind headings create a trendy, dynamic background without loading any images. Combine with filter: blur() for a soft ambient glow effect.
  • Rounded card components: 12–16px border-radius on card containers is the de facto standard in modern UI. It provides enough roundness to feel friendly without looking playful.
  • Avatar images: 50% for circular avatars, or 8–12px for the "rounded square" style used by GitHub and Slack. The rounded square approach preserves more of the image while still looking polished.
  • Pill buttons and tags: A large border-radius (9999px) turns any element into a pill shape. This is standard for primary actions, status badges, and category tags.
  • Testimonial quote backgrounds: A blob shape behind a testimonial quote adds visual interest without the formality of a rectangle. It signals "human voice" more effectively than sharp corners.
  • Loading and skeleton animations: Morphing blob shapes used as loading indicators feel more refined than spinning circles. The shape change draws the eye without being distracting.
  • Image containers with clipped corners: Applying border-radius to a container with overflow: hidden clips the image to the rounded shape—no need for object-fit hacks or SVG masks.
  • Tooltip and popover shapes: While border-radius alone cannot create pointer arrows, rounding the main body of a tooltip (6–8px) makes it look more polished and less like a browser default.

Best Practices for Border Radius in Production

  • Define a radius scale as design tokens: Create three to five named values (e.g., --radius-sm: 6px, --radius-md: 12px, --radius-lg: 16px, --radius-full: 9999px) and use them consistently. This prevents the "every card has a different radius" problem.
  • Use px for UI components, % for decorative shapes: Buttons, cards, and inputs should use pixel values for predictable, consistent rounding. Blob backgrounds and decorative elements should use percentages for organic, responsive curves.
  • Match radius to element size: A 16px radius looks appropriate on a 300px card but absurd on a 32px icon container. Scale your radius values proportionally—or use percentage values that scale automatically.
  • Pair radius with shadow for elevation: Rounded corners without a shadow look flat and incomplete. A subtle box shadow underneath a rounded card completes the "floating surface" illusion.
  • Keep blob shapes subtle: In production UI, blobs work best as background decorations with low opacity. A fully saturated, hard-edged blob as a primary UI element can look unprofessional. Blur and transparency are your friends.
  • Test on non-square elements: A border-radius that looks great on a 200×200 square may produce unexpected results on a 300×150 rectangle because percentage values calculate differently for each axis.

Common Mistakes When Using Border Radius

  • Using inconsistent radii across similar components: If your card components have 12px, 14px, 16px, and 18px radius across different pages, the UI feels subtly disorganized. Pick one value per component type and stick to it.
  • Forgetting overflow: hidden on image containers: Without overflow: hidden, the image inside a rounded container will peek out past the curves. This is one of the most common border-radius bugs.
  • Using percentage values for UI components: border-radius: 5% on a 400px-wide card produces 20px corners, but the same 5% on a 200px card produces only 10px. Use pixel values for consistent UI rounding.
  • Overusing 50% for everything: Circular avatars are fine, but circular buttons, circular cards, and circular inputs look odd. Reserve 50% for avatars and icons; use moderate pixel values for everything else.
  • Ignoring the eight-value syntax for complex shapes: If your blob looks "off" but you cannot figure out why, it may need different horizontal and vertical radii on each corner. The slash syntax (30% 70% / 70% 30%) gives you that control.
  • Not testing at different sizes: A blob shape generated at 200px preview size will look proportionally different at 80px or 500px. Always verify at the actual size used in your layout.

Performance Considerations

Border radius itself has negligible rendering cost. A static rounded element paints just as fast as a sharp-cornered one. The performance considerations arise in two specific scenarios:

  • Animating border-radius: Changing border-radius on every frame triggers a repaint because the browser must recalculate the clip region. On a page with many animated blobs, this can cause jank. Mitigation: use will-change: border-radius sparingly on the animated elements, or keep the number of simultaneous morphing blobs under three.
  • Border radius on large elements with box-shadow: A large element (500px+) with both a complex border-radius and a large blur shadow forces the browser to clip a large area and then paint a large shadow. If you have many such elements (e.g., a grid of 20+ cards), consider simplifying the shadow or using border-radius without shadow on repeated items.
  • Overflow clipping cost: overflow: hidden combined with border-radius creates a new clipping context, which can prevent sub-pixel rendering optimizations. This is rarely noticeable, but on low-end mobile devices with many clipped elements, it can contribute to scroll jank.

In practice, border-radius performance is almost never a problem. The scenarios above are edge cases that only matter on pages with dozens of animated or shadowed rounded elements. If your page also loads heavy images, run them through our image compressor first—that will have a far larger impact than any border-radius optimization.

Accessibility Notes

  • Border radius is purely visual: It does not affect keyboard navigation, screen reader output, or semantic structure. A rounded button and a sharp-cornered button are identical to assistive technology.
  • Do not rely on shape to convey meaning: If a blob shape is used as a decorative background, mark it with aria-hidden="true" so screen readers skip it. If the shape contains meaningful content, ensure the content is accessible regardless of the visual shape.
  • Morphing animations and motion sensitivity: If you animate border-radius for loading indicators or decorative blobs, wrap the animation in @media (prefers-reduced-motion: no-preference). Users who have enabled reduced motion should see a static shape instead of a morphing one.
  • Focus indicators on rounded elements: outline does not follow border-radius in all browsers. Use a box-shadow focus ring instead, which respects the element's rounded corners and provides a more polished accessible experience.
  • Color contrast on shaped backgrounds: If text sits on top of a colored blob, verify that the text-to-blob contrast ratio meets WCAG 2.1 requirements (4.5:1 for body text, 3:1 for large text). Use our color picker to verify contrast values.

Advanced Techniques

Eight-value syntax for elliptical blobs

/* Four-value: same curve on both axes */
.blob-simple {
  border-radius: 30% 70% 70% 30%;
}

/* Eight-value: independent horizontal/vertical curves */
.blob-complex {
  border-radius: 30% 70% 70% 30% / 30% 30% 70% 70%;
}
/* Before slash: horizontal radii (TL TR BR BL)
   After slash:  vertical radii (TL TR BR BL) */

The eight-value syntax produces shapes that the four-value syntax cannot achieve. The horizontal and vertical radii on each corner are independent, so you can create blobs that stretch more in one direction than the other. This is how you get the "water droplet" and "amoeba" shapes seen on high-end landing pages.

CSS variable-driven shape morphing

.shape {
  --r1: 30%; --r2: 70%; --r3: 70%; --r4: 30%;
  --r5: 30%; --r6: 30%; --r7: 70%; --r8: 70%;
  border-radius: var(--r1) var(--r2) var(--r3) var(--r4) /
                   var(--r5) var(--r6) var(--r7) var(--r8);
  transition: border-radius 0.6s ease;
}
.shape:hover {
  --r1: 70%; --r2: 30%; --r3: 30%; --r4: 70%;
  --r5: 70%; --r6: 70%; --r7: 30%; --r8: 30%;
}

By storing each radius value in a CSS variable, you can create hover-driven morphing effects without JavaScript. The transition property animates the border-radius change smoothly. Note that CSS custom property transitions require @property registration in some browsers for smooth animation.

Clipping images to blob shapes

.blob-image {
  width: 300px;
  height: 300px;
  border-radius: 30% 70% 70% 30% / 30% 30% 70% 70%;
  overflow: hidden;
}
.blob-image img {
  width: 100%;
  height: 100%;
  object-fit: cover;
}

This technique clips any image to an organic blob shape without SVG clip-paths or canvas rendering. The overflow: hidden on the container is essential—it tells the browser to clip the child image to the parent's border-radius curve. This is significantly simpler and more performant than SVG-based clipping for most use cases.

Animating Border Radius in CSS

Border-radius animation is one of the few CSS animations that genuinely feels organic. Because the property controls the shape of curves rather than position or opacity, morphing between values produces a fluid, biological motion that linear transforms cannot replicate. This is why it has become a go-to technique for loading indicators, hover effects, and ambient background animations.

  • CSS Transitions: Add transition: border-radius 0.5s ease to an element, then change the border-radius value on hover. The shape smoothly morphs between the two states. This works well for buttons that "soften" on hover or cards that become more rounded when focused.
  • Keyframe animations: Define multiple keyframes with different border-radius values for a continuous looping morph. This is the standard approach for loading spinners and background decorations.
  • Combine with rotation: Adding transform: rotate() alongside border-radius animation creates a more complex, visually interesting motion. The rotation makes the asymmetry of the shape more apparent as it turns.
  • Performance: Border-radius animation triggers repaint, not reflow, so it is relatively cheap. However, on pages with many animated elements, consider using will-change: border-radius on the animated elements only—do not apply it broadly.

Here is a production-ready morphing blob animation: 

@keyframes morph {
  0%   { border-radius: 30% 70% 70% 30% / 30% 30% 70% 70%; }
  25%  { border-radius: 58% 42% 30% 70% / 50% 60% 40% 50%; }
  50%  { border-radius: 70% 30% 30% 70% / 70% 70% 30% 30%; }
  75%  { border-radius: 42% 58% 70% 30% / 40% 30% 70% 60%; }
  100% { border-radius: 30% 70% 70% 30% / 30% 30% 70% 70%; }
}

.morphing-blob {
  width: 200px;
  height: 200px;
  background: linear-gradient(135deg, #10b981, #06b6d4);
  animation: morph 8s ease-in-out infinite;
}

Four keyframe stops create a smooth, continuous morph. The eight-value syntax ensures the shape changes along both axes. The ease-in-out timing function creates natural acceleration and deceleration at each stop.

Pixel vs Percentage: When to Use Each

  • Pixel values (px): Create a fixed-radius curve regardless of element size. A border-radius: 12px always produces a 12-pixel curve. This is ideal for UI components like buttons, cards, and inputs where you want consistent corner sizes across different element sizes.
  • Percentage values (%): Scale relative to the element's width and height. A border-radius: 50% on a square element creates a perfect circle, but on a rectangle it creates an ellipse. This is essential for organic blob shapes because the curves adapt to the element's proportions.
  • Mixing units: You can mix px and % in the same declaration (e.g., border-radius: 12px 50% 12px 50%), though this is uncommon and the results can be hard to predict on different element sizes.
  • Responsive design: Percentage values are better for responsive layouts because they scale with the element. Pixel values stay fixed, which can look inconsistent on different screen sizes unless you use responsive units like rem or clamp values.

This generator uses percentage values because they produce the most visually interesting and responsive blob shapes. For standard UI components with uniform rounded corners, pixel values are typically the better choice.

Frequently Asked Questions

What is border-radius in CSS?

The border-radius CSS property rounds the corners of an element's outer border edge. It can accept one to four values, allowing you to control each corner independently to create everything from slightly rounded rectangles to perfect circles and complex organic shapes.

How to create blob shapes in CSS?

Set unequal border-radius percentage values on all four corners. For example, border-radius: 30% 70% 70% 30% creates an organic blob. Percentage values scale relative to the element's dimensions, producing smooth asymmetric curves that look natural and fluid.

What values are best for organic shapes?

Use percentage values between 20% and 80% with contrasting pairs. Good starting points: 30% 70% 70% 30% for a leaf, 60% 40% 30% 70% for a blob, or 50% 50% 50% 50% for a circle. Avoid values near 0% or 100% on all corners as they produce rectangles or circles rather than organic forms.

Can I use border-radius for UI design?

Absolutely. Border-radius is used for rounded buttons, card containers, avatar images, notification badges, and decorative background shapes. Organic blobs created with border-radius are popular as hero section accents and floating UI elements in modern web design.

Is this border radius generator free?

Yes, completely free with no signup required. Open the page, adjust the sliders, and copy the CSS code directly into your project. No paywalls, accounts, or usage limits.

What is the difference between px and % in border-radius?

Pixel values create fixed-radius corners that stay the same regardless of element size—ideal for buttons and cards. Percentage values scale relative to the element's dimensions, which is essential for creating responsive organic blob shapes that maintain their proportions at any size.

Does border-radius work in all browsers?

Yes, border-radius is fully supported in all modern browsers including Chrome, Firefox, Safari, Edge, and Opera. No vendor prefixes are needed. It has been universally supported since 2013.

Can I animate border-radius with CSS?

Yes, border-radius can be animated using CSS transitions or keyframe animations. You can create smooth morphing effects by transitioning between different border-radius values—popular for loading indicators, hover effects, and decorative animated backgrounds.

How do I make a perfect circle with border-radius?

Set border-radius to 50% on a square element (equal width and height). If the element is not square, 50% will create an ellipse instead. To force a circle on a non-square element, use a pixel value equal to half the smaller dimension.

Can border-radius clip content inside an element?

Yes. By default, border-radius clips the element's background and border to the rounded shape. Add overflow: hidden to also clip child elements like images. This is how rounded avatar images work—the image is clipped to the parent's border-radius curve.

What is the eight-value syntax for border-radius?

Border-radius supports eight values separated by a slash: border-radius: 10px 20px 30px 40px / 50px 60px 70px 80px. The first four values set horizontal radii, and the values after the slash set vertical radii. This lets you create elliptical curves on each corner independently for more complex shapes.

Does border-radius affect box-shadow?

Yes, box-shadow follows the border-radius of the element. A rounded element will have a rounded shadow that matches its shape. For best results, pair rounded corners with appropriately styled shadows using a box shadow generator.

How do I make only some corners rounded?

Use individual corner properties: border-top-left-radius, border-top-right-radius, border-bottom-right-radius, border-bottom-left-radius. Set the corners you want rounded to your desired value and leave the others at 0. You can also use the four-value shorthand where 0 means no rounding on that corner.

Why does my border-radius look different on rectangles vs squares?

Percentage-based border-radius is calculated relative to both the width and height of the element independently. On a rectangle, 50% of the width differs from 50% of the height, so the corners become elliptical rather than circular. This is by design and is what makes blob shapes possible on non-square elements.

Practical Guide to CSS Border Radius Generator

This guide covers how frontend developers and designers use a border radius generator as part of a real production workflow—not as an isolated toy, but as a calibration step in a component styling pipeline that produces consistent, maintainable results across an entire project.

Workflow: From Blank Element to Styled Component

Most developers reach for a border radius generator at a specific point in the component-building process: the element has its layout, padding, and content, but it looks like a plain rectangle and needs visual finishing. The typical workflow goes like this: open the generator, set the size slider to match your element's approximate dimensions, drag the corner sliders until the shape matches your intent, note whether the shape holds up when you change the preview size, copy the value, and paste it into your CSS as a variable.

The critical step that most people skip is the size test. A blob that looks organic at 200px can look like a distorted oval at 80px or a nearly circular form at 500px. Always test at multiple sizes before committing to a value, especially if the element is responsive.

Professional Tips

  • Generate blob values at the actual render size. If your hero blob will be 600px wide, set the size slider to 600. The percentage curves will distribute differently at different sizes, and the shape you see at 200px is not the shape you get at 600px.
  • Save shape values as named constants. Instead of pasting raw values into components, save them as CSS variables with descriptive names: --shape-hero-blob: 30% 70% 70% 30% / 30% 30% 70% 70%;. This makes your shapes searchable, documentable, and reusable.
  • Use the Random button as a brainstorming tool. When you are stuck in a creative rut, hit Random 10-20 times and screenshot any shapes that catch your eye. You can refine them afterward with manual slider adjustments.
  • Combine border-radius with CSS filters for complex effects. A blob shape with filter: blur(40px) becomes an ambient glow. With filter: contrast(1.5), the edges sharpen into something more crystalline. The shape is just the starting point.

Team Collaboration

In a team, the biggest risk with border-radius is drift. Developer A uses 12px for cards, Developer B uses 14px, and by the end of the quarter every card has a slightly different roundness. The fix is to agree on radius tokens early—use this generator to define the exact values for each token level, document them in your design system, and enforce them in code review.

For blob shapes specifically, the risk is that every designer picks a different blob for every page, creating visual inconsistency. Agree on two to three blob shapes for the entire project (e.g., one for hero sections, one for feature highlights, one for testimonial backgrounds) and reuse them. This creates a coherent visual language rather than a collection of unrelated organic shapes.

Real Projects

  • SaaS landing pages: A single blurred blob behind the hero heading is the most common use case. Generate the shape, apply a gradient fill, add blur and low opacity, and position it absolutely behind the text. Total CSS: about eight lines.
  • Design system documentation: Include the actual blob shapes in your design system docs so developers can copy the exact values without opening the generator. Screenshot each shape alongside its CSS value.
  • Portfolio sites: Blob-shaped image containers for project thumbnails add personality without requiring custom SVG paths. Use overflow: hidden to clip the images.
  • Mobile app web wrappers: Rounded corners on web-based mobile UI components need to match native platform conventions (typically larger radii on iOS, smaller on Android). Generate the exact values for each platform and store them as platform-specific tokens.

Mistakes That Cost Time

  • Generating shapes at the wrong size. A blob designed at 150px preview will look noticeably different when rendered at 400px. Always match the preview size to your target size.
  • Not saving values immediately. If you generate a perfect shape, use it in one component, and then close the tab, you will spend ten minutes trying to recreate it. Paste values into your CSS variables file the moment you generate them.
  • Using blob shapes as primary UI containers. Blobs work as background decorations, not as card or button shapes. A blob-shaped card looks unprofessional and makes layout alignment difficult. Keep blobs in the background layer.
  • Ignoring dark mode when choosing blob colors. A pastel blob that looks subtle on white can look garish on a dark background. Always check your blob colors in both themes.

Productivity Tips

  • Keep this generator open in a pinned tab during design work. The friction of opening a new tab is small but real—having it ready removes the excuse to "just type a value."
  • If you use Figma, generate the border-radius values here first, then apply them in Figma using the corner radius inputs. This gives you pixel-perfect consistency between design and code.
  • Create a small snippet file with your most-used radius values so you can paste them without opening any tool for standard components.
  • When reviewing pull requests, flag any hardcoded border-radius values that are not referencing design tokens. This is the same discipline you apply to colors and shadows—radius values should be tokenized too.
